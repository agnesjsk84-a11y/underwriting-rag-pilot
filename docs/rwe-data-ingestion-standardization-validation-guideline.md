# RWE 데이터 적재·표준화·검증 가이드

> 기반 문서: 건강보험심사평가원(HIRA), 「약제성과평가를 위한 실제 근거(RWE) 생성 가이드라인」, V1.0, 2026.06.
>
> 문서 목적: 실제 자료(RWD)를 플랫폼에 적재하고 표준화할 때 적용할 운영 기준을 정의하고, 검증 단계에서 판단 근거·체크리스트·증빙자료를 일관되게 관리한다.

## 1. 문서의 성격과 적용 원칙

이 문서는 원문 가이드라인 전체를 그대로 옮긴 전사본이 아니라, 데이터 플랫폼의 구축·운영·검증에 사용할 수 있도록 핵심 내용을 실무 기준으로 재구성한 문서다.

- **원문 기준**: HIRA 가이드라인에 명시된 원칙과 고려사항
- **플랫폼 운영 기준**: 원문을 실제 시스템에 적용하기 위해 구체화한 절차, 상태값, 승인 게이트 및 산출물
- **프로젝트별 기준값**: 결측률, 허용 오차, 중복률 등 연구 특성에 따라 사전에 확정해야 하는 수치

이 문서는 규제기관의 공식 승인이나 법률 자문을 대신하지 않는다. 개인정보 처리, 가명정보 결합, 보존기간 및 외부 제공은 적용 시점의 법령·기관 정책·IRB 승인 내용을 별도로 확인해야 한다.

## 2. 적용 범위

다음 데이터와 처리 단계에 적용한다.

- 1차 자료: eCRF, 환자보고결과, 의료진 입력, 연구 전용 앱·기기 데이터
- 2차 자료: 전자건강기록(EHR), 보험청구자료, 레지스트리, CDW, 사망자료 등
- 정형 데이터와 비정형 텍스트·영상에서 추출한 구조화 데이터
- 단일 자료원 및 여러 자료원을 결합한 연계 데이터
- 원천 데이터 수신부터 분석용 데이터셋 확정까지의 수집, 적재, 정제, 변환, 표준화, 연계, 검증, 배포 및 보존

## 3. 핵심 개념

| 용어 | 운영 정의 |
|---|---|
| RWD | 일상적인 보건의료 환경에서 생성·수집되는 환자 건강상태 및 의료 제공 관련 자료 |
| RWE | RWD를 적절한 연구설계와 분석방법으로 분석해 생성한 임상적 근거 |
| 데이터 거버넌스 | 정책, 규칙, 품질 기준, 권한 및 책임을 정하고 준수하는 체계 |
| 데이터 관리 | 데이터의 수집, 운영, 저장, 변환, 검토, 보존을 실행하는 활동 |
| 데이터 큐레이션 | 분석 목적에 맞는 데이터셋을 만들기 위해 품질관리, 표준화, 의미 일관성 및 추적성을 확보하는 전 주기 활동 |
| ETL | 원천에서 데이터를 추출하고, 표준 구조와 값으로 변환한 뒤, 목표 저장소에 적재하는 과정 |
| 목적 적합성 | 데이터가 특정 연구질문에 답할 만큼 신뢰할 수 있고 관련성이 있는 상태 |
| 감사추적 | 데이터의 최초 값과 변경 전후 값, 변경자, 시각, 대상 필드 및 변경 사유를 보존한 기록 |

## 4. 품질관리 기본 원칙

원문은 데이터 품질을 세 가지 관점으로 구분한다.

1. **생성·처리 과정의 품질**: 오류 발견·수정, 검증, 입출력, 표준 프로세스가 적절한가?
2. **데이터셋 자체의 품질**: 결측, 비현실적 값, 중복, 코드·형식 불일치가 관리되는가?
3. **연구질문에 대한 목적 적합성**: 필요한 변수, 기간, 표본 수, 대표성, 완전성이 충분한가?

플랫폼은 다음 원칙을 기본 통제로 적용한다.

- 주요 변수의 정의와 검증 규칙은 적재 전에 승인한다.
- 원천 데이터는 변경하지 않고 읽기 전용으로 보존한다.
- 원천부터 최종 분석 데이터까지 레코드·변수·코드의 계보를 추적할 수 있어야 한다.
- 자동 변환과 수동 수정 모두 재현 가능하고 감사 가능해야 한다.
- 정제 과정에서 삭제·대체된 값은 원본과 사유를 함께 보존한다.
- 데이터 품질 문제와 연구 해석에 미치는 영향을 숨기지 않고 보고한다.
- 검증 실패 데이터는 승인 없이 분석·서비스 단계로 전달하지 않는다.

## 5. 역할과 책임

| 역할 | 주요 책임 | 필수 승인 또는 산출물 |
|---|---|---|
| 데이터 소유자 | 이용 범위, 제공 조건, 보존·폐기 정책 결정 | 이용 승인, 제공 명세 |
| 연구책임자 | 연구질문, 주요 변수, 분석 적합성 결정 | 연구계획서, 변수 정의 승인 |
| 데이터 스튜어드 | 데이터 사전, 표준 용어, 품질 기준 관리 | 데이터 사전, 매핑 승인 |
| 데이터 엔지니어 | 수신, ETL, 표준화, 적재 및 기술 검증 | ETL 명세, 실행 로그, 조정 보고서 |
| 데이터 품질 담당자 | 품질 규칙, 프로파일링, 쿼리 및 결함 관리 | 품질 보고서, 결함 종료 승인 |
| 통계·역학 담당자 | 목적 적합성, 비뚤림, 결측 및 민감도 평가 | 분석 적합성 평가 |
| 개인정보·보안 담당자 | 가명처리, 접근통제, 전송·저장·보존 점검 | 보안·개인정보 검토 기록 |
| 최종 승인자 | 검증 결과와 잔여 위험을 검토해 배포 결정 | 승인, 조건부 승인 또는 반려 |

한 사람이 여러 역할을 수행할 수 있으나, 운영 환경에서는 ETL 실행자와 최종 승인자의 분리를 권장한다.

## 6. 단계별 운영 기준과 승인 게이트

### Gate 0. 연구질문 및 데이터 요구사항 확정

적재 전에 다음을 확정한다.

- 목표 모집단, 치료군·대조군, 결과변수, 공변량·교란요인, 하위그룹
- 인덱스 날짜, 사전 관찰기간, 추적관찰기간, 데이터 마감일
- 필수·선택 변수 및 허용 가능한 대리 변수
- 예상 데이터 규모, 갱신 주기 및 사용 목적
- 프로젝트별 품질 임계값과 예외 승인 절차

**통과 증빙**: 승인된 연구계획서, 데이터 요구사항 명세서, 주요 변수 목록

### Gate 1. 자료원 목적 적합성 평가

자료원별로 다음을 기록한다.

- 자료원 이름, 버전, 소유자, 웹사이트, 데이터 마감일·추출일
- 원래 수집 목적과 자료원 유형
- 수집 환경, 지역, 기간, 모집단 범위, 대표성
- 데이터 수집·코딩 방식과 시간에 따른 변경
- 원시 데이터 접근 가능 여부 및 제공자의 선행 정제·변환 내용
- 데이터 모델, 데이터 사전, 품질보증 문서의 제공 여부
- 연계 데이터셋, 연계 방식·성능·주체
- 과거 활용 이력, 알려진 한계 및 잠재적 선택 비뚤림

**판정 기준**

- 관련성: 필요한 모집단·치료·결과·공변량을 정의할 수 있는가?
- 신뢰성: 생성·관리 절차와 완전성·정확성을 확인할 수 있는가?
- 충분성: 표본 크기와 추적기간이 연구질문에 답하기에 충분한가?
- 일반화 가능성: 표본과 진료환경이 목표 모집단·환경을 반영하는가?

**통과 증빙**: 자료원 평가서, DataSAT 기반 평가표, 이용 승인 문서

### Gate 2. 수신 및 원천 보존

- 전송 경로, 암호화, 파일 형식, 문자 인코딩, 압축 방식 및 수신 주기를 사전에 정의한다.
- 수신 파일의 이름, 크기, 행 수, 생성·수신 시각, 해시값을 매니페스트로 기록한다.
- 예상 파일·테이블·레코드 수와 실제 수신 결과를 대조한다.
- 원천 영역은 읽기 전용으로 보존하고, 재처리는 동일 원천 스냅샷을 기준으로 수행한다.
- 가명처리 또는 익명화 적용 여부와 처리 주체·방법을 기록한다.
- 전송 실패·중복 수신·부분 수신을 자동 탐지한다.

**통과 증빙**: 수신 매니페스트, 해시 검증 결과, 원천 스냅샷 ID, 전송 로그

### Gate 3. 프로파일링과 데이터 사전 확정

데이터 사전에는 최소한 다음 항목을 포함한다.

| 구분 | 필수 내용 |
|---|---|
| 식별 | 원천 변수명, 표준 변수명, 테이블, 고유 ID |
| 의미 | 업무 정의, 임상적 정의, 조작적 정의, 산출 근거 |
| 형식 | 자료형, 길이, 날짜 형식, 단위, 허용 정밀도 |
| 값 | 허용값, 코드 체계·버전, 정상·허용 범위 |
| 시간 | 수집 시점, 기준일과의 관계, 시간가변 여부 |
| 출처 | 자료원, 생성 시스템, 입력·산출 주체 |
| 필수성 | 필수·선택 구분, 조건부 입력 규칙 |
| 품질 | 검증 규칙, 결측 처리, 오분류 가능성, 검증 방법 |
| 이력 | 최초 적용일, 변경일, 변경 사유, 승인자 |

프로파일링은 행 수, 유일값, 결측률, 분포, 최소·최대, 코드 빈도, 날짜 범위, 중복률을 포함한다.

**통과 증빙**: 승인된 데이터 사전, 원천 프로파일 보고서, 검증 규칙 목록

### Gate 4. 정제

정제 대상은 다음과 같다.

- 필수값 누락과 예상치 못한 결측
- 생리적·업무적으로 불가능하거나 범위를 벗어난 값
- 동일 환자·사건의 중복 및 여러 기관에 분산된 기록
- 변수 간 논리 모순과 시간 순서 오류
- 코드 체계에 존재하지 않거나 폐기된 코드
- 비정상적인 기관별·기간별 분포 변화
- 수동 전사 오류 및 자동 추출 오류

정제 시 자동 규칙과 수동 쿼리 검토를 병행할 수 있다. 삭제·수정·대체는 규칙 ID, 원본값, 변경값, 사유, 실행 버전과 함께 기록한다. 해결할 수 없는 오류는 임의로 보정하지 않고 결함 또는 제한사항으로 남긴다.

**통과 증빙**: 정제 규칙서, 쿼리 리포트, 전후 행 수 조정표, 미해결 결함 목록

### Gate 5. 변환 및 표준화

표준화는 다음 순서로 수행한다.

1. 구조 표준화: 목표 데이터 모델의 테이블·키·관계로 변환
2. 형식 표준화: 자료형, 날짜, 시간대, 인코딩, 숫자 정밀도 통일
3. 단위 표준화: 원 단위와 변환계수 보존, 목표 단위로 환산
4. 용어 표준화: 원천 코드와 표준 코드의 매핑 및 버전 관리
5. 의미 표준화: 동일 개념의 정의·분류·측정시점 일치
6. 파생변수 생성: 알고리즘, 가정, 입력 변수, 검증 결과 기록

가이드라인에서 제시한 표준 예시는 CDISC, ICD-10, SNOMED, MedDRA, LOINC, RxNorm이다. 실제 적용 표준과 버전은 데이터 유형과 연구 목적에 맞게 사전 결정한다.

**표준화 검증 기준**

- 모든 표준 변수는 원천 필드 또는 승인된 산출 규칙으로 역추적 가능해야 한다.
- 미매핑, 일대다·다대일 매핑 및 의미 손실을 별도 보고한다.
- 자동 전송·변환 결과에 대해 레코드 수와 핵심값을 원천과 조정한다.
- 변환 후에도 정확성, 대표성 및 목적 적합성이 유지되는지 확인한다.
- NLP·AI 알고리즘은 데이터 출처, 모델·알고리즘 버전, 가정, 매개변수 및 검증 지표를 기록한다.

**통과 증빙**: 소스-타깃 매핑서, 표준 용어 매핑표, ETL 코드 버전, 테스트 결과, 미매핑 보고서

### Gate 6. 자료 연계

자료 연계 시 다음을 문서화한다.

- 연계 목적, 대상 데이터셋, 연계 단위 및 제공 주기
- 확정적 또는 확률적 연계 방식과 사용 변수
- 동의 필요 여부, 법적 근거, 계약, 책임 및 접근 정책
- 연계 전후 대상자 수와 레코드 수
- 미연계·불일치 비율과 시간에 따른 연계 성능
- 오연계·미연계 가능성 및 대표성에 미치는 영향
- CDM 적용 여부, 정보 손실 및 자료원 간 코딩 이질성

확률적 연계는 임계값 선택 근거와 강건성 평가를 포함해야 한다.

**통과 증빙**: 연계 명세, 성능 평가, 단계별 대상자 흐름, 개인정보 검토 기록

### Gate 7. 품질 검증 및 목적 적합성 재평가

검증은 최소한 다음 차원을 포함한다.

| 품질 차원 | 확인 질문 | 대표 지표 |
|---|---|---|
| 완전성 | 필요한 값과 레코드가 존재하는가? | 결측률, 수신율, 필수 필드 충족률 |
| 정확성 | 데이터가 실제 상태 또는 신뢰 가능한 기준을 반영하는가? | 원본 일치율, 양성예측도, 민감도·특이도 |
| 유효성 | 형식·범위·코드 규칙을 충족하는가? | 규칙 통과율, 유효 코드율 |
| 일관성 | 변수·테이블·기관·시점 간 모순이 없는가? | 참조무결성 오류율, 논리 오류율 |
| 유일성 | 중복 환자·사건이 통제되는가? | 중복률, 키 충돌 수 |
| 적시성 | 마감일과 갱신 주기가 목적에 적합한가? | 데이터 지연 일수, 최신성 |
| 추적성 | 결과를 원천과 처리 이력으로 재현할 수 있는가? | 계보 연결률, 감사로그 완전성 |
| 관련성 | 연구에 필요한 정보와 충분한 표본·기간이 있는가? | 변수 가용률, 표본 수, 추적기간 |

주요 연구변수별로 목표 개념, 조작적 정의, 유효한 수집의 최소 기준, 타당성 평가방법·결과, 결측률을 기록한다. 최소 대상은 연구 모집단, 치료군, 대조군, 1·2차 결과변수, 추적관찰기간, 교란요인 및 하위그룹이다.

**통과 증빙**: 품질 스코어카드, 주요 변수 타당성 평가표, 목적 적합성 재평가, 잔여 위험 목록

### Gate 8. 보안·개인정보·감사추적 검증

- 직무 기반 최소 권한과 고유 사용자 계정을 적용한다.
- 반복 로그인 실패, 무단 접근 및 권한 변경을 탐지한다.
- 데이터 전송·저장 시 기밀성과 무결성을 보호한다.
- 전자 서명과 감사추적의 작동 여부를 검증한다.
- 감사추적은 일반 사용자가 비활성화·수정·삭제할 수 없어야 한다.
- 실시간·오프사이트 백업과 복구 계획을 수립하고 복구 테스트를 기록한다.
- 개인정보 처리 목적, 동의 또는 법적 근거, IRB·기관 승인 및 제3자 제공 조건을 문서화한다.
- 보존기간이 여러 규정에서 다르면 적용 근거를 검토해 더 긴 기간을 원칙으로 한다.

**통과 증빙**: 접근권한 검토표, 보안 점검 결과, 감사추적 샘플, 백업·복구 시험 결과, 개인정보 처리 근거

### Gate 9. 배포 승인과 지속 모니터링

최종 판정은 다음 세 상태로 관리한다.

- **승인**: 필수 검증을 통과하고 중대한 미해결 결함이 없음
- **조건부 승인**: 목적에 제한을 두면 사용 가능하며, 제한사항·완화조치·책임자가 승인됨
- **반려**: 데이터 무결성, 개인정보, 주요 변수 또는 목적 적합성에 중대한 결함이 있음

배포 후에는 갱신 배치마다 핵심 품질지표, 분포 변화, 코드 체계 변경, 데이터 지연, 미매핑률 및 접근 로그를 모니터링한다.

**통과 증빙**: 릴리스 승인서, 데이터셋 버전, 알려진 제한사항, 모니터링 대시보드

- [ ] 가명처리·익명화 적용 여부와 처리 결과가 기록되었다.

- [ ] 표준 구조, 형식, 단위, 용어 및 의미 매핑이 적용되었다.
- [ ] 표준 코드 체계와 버전이 명시되었다.
- [ ] 미매핑과 일대다·다대일 매핑이 검토되었다.
- [ ] 단위 변환식과 원 단위가 보존되었다.
- [ ] 파생변수 알고리즘, 입력값, 가정 및 버전이 문서화되었다.

## 8. 검증 판정 기준

### 8.1 중대 결함 예시

다음은 원칙적으로 배포를 차단한다.

- 데이터 수신 무결성 또는 원천 스냅샷을 확인할 수 없음
- 주요 결과·노출·모집단 변수의 정의 또는 계보가 없음
- 개인정보 처리 근거, 접근통제 또는 감사추적에 중대한 위반이 있음
- ETL 전후 대상자·레코드 수 차이를 설명할 수 없음
- 코드·단위 변환 오류가 주요 분석 결과에 영향을 줄 수 있음
- 재실행 결과가 일치하지 않음
- 자료원이 연구질문에 필요한 변수·기간·표본을 제공하지 못함

### 8.2 프로젝트별 수치 기준 템플릿

HIRA 원문은 모든 프로젝트에 동일하게 적용할 고정 임계값을 제시하지 않는다. 다음 값은 연구·데이터 특성에 따라 사전 승인하여 사용한다.

| 지표 | 기준값 | 산출 단위 | 초과 시 조치 | 승인자 |
|---|---:|---|---|---|
| 필수 변수 결측률 | `[프로젝트 정의]` | 변수·기관·기간별 | 원인 분석, 보완 또는 사용 제한 |  |
| 중복 환자/사건 비율 | `[프로젝트 정의]` | 데이터셋별 | 중복 제거 규칙 검토 |  |
| 유효 코드 매핑률 | `[프로젝트 정의]` | 코드 체계별 | 미매핑 검토·예외 승인 |  |
| 범위·논리 오류율 | `[프로젝트 정의]` | 규칙별 | 쿼리·정제 후 재검증 |  |
| 원천-목표 레코드 조정률 | `[프로젝트 정의]` | 테이블별 | 차이 원인 설명·반려 |  |
| 자료 연계 성공률 | `[프로젝트 정의]` | 자료원 조합별 | 대표성·비뚤림 평가 |  |
| 데이터 최신성 | `[프로젝트 정의]` | 지연 일수 | 배치 중단 또는 경고 |  |

## 10. 플랫폼 화면에서 확인할 세부 사항

### 10.1 데이터셋 개요

- 데이터셋 ID, 이름, 소유자, 자료원, 버전, 마감일, 추출일
- 사용 목적, 승인 상태, 허용된 사용 범위
- 모집단 수, 레코드 수, 기간, 갱신 주기
- 적용 데이터 모델과 표준 용어 버전
- 품질 점수와 중대 결함 수

### 10.2 파이프라인 실행 이력

- 원천 스냅샷 ID와 해시
- 실행 ID, 코드·설정 버전, 실행자, 실행 시각
- 단계별 입력·출력 건수와 실패 건수
- 재시도·롤백 여부와 실패 사유
- 이전 배치 대비 분포·결측·매핑률 변화

### 10.3 규칙별 검증 결과

- 규칙 ID, 설명, 적용 테이블·변수
- 중요도, 임계값, 실제값, 판정
- 실패 레코드 수와 샘플
- 원인, 담당자, 조치, 재검증 결과
- 예외 승인자와 만료일

### 10.4 계보와 감사

- 표준 변수에서 원천 변수·파일·레코드까지 역추적
- 변환식, 단위 변환, 코드 매핑, 파생 알고리즘
- 데이터 변경 전후 값, 변경자, 시각, 사유
- 문서·ETL·데이터셋 버전 간 연결

## 11. 변경 관리

변경은 내부 변경과 외부 변경으로 구분한다.

- 내부 변경: CRF 필드, 데이터 정의, 검증 규칙, ETL, 워크플로우 수정
- 외부 변경: 치료·진료환경 변화, 자료원 구조·코딩 체계 변경, 소프트웨어 업데이트, 인력 교체

모든 변경은 다음 절차를 따른다.

1. 변경 요청과 사유 기록
2. 데이터 품질·연구결과·보안·일정에 미치는 영향 평가
3. 운영위원회 또는 승인자의 사전 승인
4. 데이터 사전, SOP, 매핑서, 코드, 교육자료 업데이트
5. 검증 환경에서 회귀 테스트
6. 적용일·버전·대상 기관 공지
7. 운영 반영 후 모니터링
