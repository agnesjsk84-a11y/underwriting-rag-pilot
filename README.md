# 🏥 언더라이팅 RAG Assistant — 파일럿 프로젝트

> 보험 언더라이터를 위한 의료 문서 기반 RAG(Retrieval-Augmented Generation) 질의응답 시스템

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **목적** | 의료 문서를 벡터 DB로 구축하고 LLM으로 언더라이팅 관련 질의응답 제공 |
| **배경** | RAG를 활용한 지식 플랫폼 구축 |
| **기간** | 2026년 7월 |
| **개발 환경** | Google Colab (무료) |
| **최종 스택** | Groq API + LangChain + FAISS + ipywidgets |

## 프로젝트 배경

보험 언더라이팅 현장에서는 방대한 의료 문헌과 가이드라인을 빠르게 검색하고 판단 근거를 정리하는 작업이 반복된다. 이 파일럿은 다음 질문에서 출발했다:

> **"RAG와 LLM을 활용하면 언더라이터의 의사결정을 얼마나 효율적으로 지원할 수 있을까?"**

임상 간호사 → 개인보험 언더라이터 → 재보험 언더라이터로 이어지는 경력에서 축적한 도메인 지식을 직접 시스템 설계에 반영하여, 기술자가 혼자서는 만들기 어려운 **의료 특화 지식 플랫폼**을 구현했다.

## 기술 스택

```
LLM            : Llama 3.3 70B (Groq API — 무료)
임베딩         : jhgan/ko-sroberta-multitask (한국어 특화)
벡터 DB        : FAISS (로컬)
오케스트레이션 : LangChain LCEL
UI             : ipywidgets (Google Colab 내장)
문서 로더      : PyPDFLoader
```

| 대장암 조기진단 및 치료 동향 | 암 진단·치료 단계별 위험 평가 |

## 인사이트

**1. 지식의 구조화가 RAG 품질의 선결 조건**

단순히 PDF를 청크로 분할하는 것만으로는 검색 정확도가 낮았다. GLP-1 질문에 알츠하이머 문서가 검색되는 문제는 프롬프트 개선으로 해결되지 않았고, 검색 단계 자체를 재설계해야 했다. 문서를 벡터 DB에 넣기 전에 **"어떤 지식을 어떻게 분류할 것인가"** 를 먼저 설계해야 한다.

**2. 메타데이터 설계의 중요성**

현재 구조는 파일명 키워드에 의존한 단순 필터링이다. 실제로는 각 청크에 `질환명`, `치료 단계`, `위험 등급`, `출처 신뢰도` 등의 메타데이터를 사전에 태깅해야 정교한 검색이 가능하다.

**3. 버전 관리의 필요성**

의료 가이드라인은 매년 업데이트된다. 동일 질환에 대해 2021년 기준과 2023년 기준이 혼재되면 답변의 신뢰도가 무너진다. 문서의 발행 연도, 개정 이력을 메타데이터로 관리하고 최신 버전 우선 검색 로직이 반드시 필요하다.

**4. 맥락(Context)의 데이터화가 핵심 과제**

언더라이팅에서 동일한 HbA1c 8%도 — 유병기간 1년차인지 10년차인지, 합병증 동반 여부, 나이, 동반 약물에 따라 판단이 완전히 달라진다. 이 맥락을 어떻게 데이터로 구조화하고 검색에 반영할 것인가가 단순 RAG와 실무 활용 가능한 RAG의 차이를 만든다. 현재 파일럿은 이 맥락을 프롬프트에만 위임했고, 이것이 가장 큰 한계이자 다음 과제다.
